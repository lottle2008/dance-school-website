## 1. 使用的系统与方式
当前项目未引入专业的日志框架（如 Winston、Pino 等），而是采用 **原生 `console` API** 作为主要的日志输出方式。这是一种在 Next.js 小型项目或 MVP 阶段常见的轻量级做法，依赖 Vercel 或 Node.js 运行时的标准输出捕获机制。

## 2. 关键文件与位置
核心日志逻辑集中在服务端 API 路由中：
- `app/api/booking/route.ts`：这是目前唯一显式包含日志输出的文件。
  - **业务日志**：使用 `console.log` 记录新产生的试听预约数据（第 55 行）。
  - **错误日志**：使用 `console.error` 捕获并记录预约提交过程中的异常（第 66 行）。

## 3. 架构与约定
- **无结构化日志**：目前的日志输出为简单的字符串拼接或对象直接打印，缺乏统一的字段结构（如 `timestamp`, `level`, `traceId` 等）。
- **手动标记**：开发者通过在 `console.log` 中添加前缀（如 `[新试听预约]`）来区分日志来源，这是一种非正式的过滤约定。
- **错误处理模式**：在 `try...catch` 块中捕获错误后，仅通过 `console.error` 输出到控制台，并向客户端返回通用的“服务器错误”提示，未集成错误追踪服务（如 Sentry）。

## 4. 开发者应遵循的规则
- **生产环境迁移**：由于内存存储和 `console` 日志在分布式环境（如 Vercel Serverless）中难以持久化和检索，建议在正式上线前迁移至支持结构化输出的日志库（如 `pino`）或接入可观测性平台。
- **敏感信息脱敏**：目前在 `console.log` 中直接打印了包含家长姓名、手机号等隐私信息的 `record` 对象。在处理用户数据时，务必对敏感字段进行脱敏处理后再输出日志。
- **统一日志层级**：建议明确 `console.log`（信息）、`console.warn`（警告）和 `console.error`（错误）的使用场景，避免在生产环境中输出过多的调试信息。
