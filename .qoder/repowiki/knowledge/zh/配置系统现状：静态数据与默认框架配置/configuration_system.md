## 1. 系统/方法概述
该项目目前**没有建立独立的配置管理系统**。配置管理主要依赖以下两种极简方式：
- **静态常量导出**：业务核心数据（如校区信息、课程列表、师资介绍）直接硬编码在 `lib/data.ts` 文件中，通过 TypeScript 对象导出供全项目引用。
- **Next.js 默认配置**：`next.config.ts` 保持为空对象，未定义任何自定义环境变量映射、重定向规则或安全策略。

## 2. 关键文件与位置
- **`lib/data.ts`**：项目的“事实配置中心”。包含了学校名称、联系方式、两个校区的详细地址与电话、所有课程的年龄段与描述等。任何业务信息的修改都需要直接编辑此文件并重新构建项目。
- **`next.config.ts`**：Next.js 的入口配置文件，目前处于初始状态，未参与运行时配置的加载。
- **`app/api/booking/route.ts`**：预约接口逻辑，其中包含了一些硬编码的业务规则（如手机号正则校验），且明确标注了数据存储目前为内存级（MVP阶段），未接入外部数据库配置。

## 3. 架构与约定
- **编译时配置（Build-time Configuration）**：由于缺乏 `.env` 文件或动态配置加载器，所有的“配置”实际上都是源代码的一部分。这意味着修改配置等同于修改代码，必须经过完整的 `build` 流程才能生效。
- **无环境分层**：项目中未发现 `.env.development` 或 `.env.production` 文件，也没有针对开发、测试、生产环境的差异化配置逻辑。
- **零外部依赖**：未引入如 `dotenv`、`zod` 或专门的配置管理库，完全依赖 Next.js 和 TypeScript 的原生能力。

## 4. 开发者应遵循的规则
- **数据修改规范**：若需更新课程价格、校区电话或教师简介，请直接修改 `lib/data.ts` 中的对应导出对象。
- **环境变量预留**：虽然当前未使用，但若未来需要接入企业微信通知（如 `route.ts` 中 TODO 所述）或数据库，应按照 Next.js 规范在项目根目录创建 `.env.local` 文件，并通过 `process.env.NEXT_PUBLIC_...`（前端可用）或 `process.env.DB_URL`（仅服务端可用）进行访问。
- **避免硬编码扩散**：目前业务逻辑（如手机号校验）散落在 API 路由中。建议随着项目迭代，将这类规则也抽取到 `lib` 目录下的独立配置或工具函数中，以保持 `data.ts` 专注于内容数据。
